20× Faster Quoting: Computer Vision + Operations Research

Moovez (Last-Mile Delivery) • 2024 to Present • Now commercialized as Quotely SaaS
Challenge

Manual job quoting took dispatchers 1 hour per estimate (phone calls, visual guesswork, spreadsheet arithmetic). Quotes were unreliable, creating billing disputes and unpredictable crew scheduling.

Business impact: Support burden scaled linearly with volume, creating a scaling bottleneck.

Solution

Built three interconnected AI systems:

  • Computer Vision: Customers photograph items, CV model estimates volume, weight class, and handling complexity
  • Operations Research: Predictive framework solving three NP-hard sub-problems (job duration, optimal crew size, 3D bin packing for truck loading)
  • Conversational AI: SMS agent handling booking confirmations, reschedules, and support queries
Technical Stack

R (Bayesian modeling), SQL, Computer Vision algorithms, Conversational AI frameworks, SMS API integration, Historical job data analysis, Predictive modeling, Operations research optimization

My Role

Engineered the OR algorithm end-to-end as a single contributor: mined historical job data, defined predictive framework, rapid prototyping and validation, drove engineering collaboration for production deployment.

Results
Quote generation: 1 hour to 3 minutes (20× faster)
  • 85% of jobs finish within quoted time window (vs. unmeasured high variance before)
  • ~50% of support tickets deflected by SMS agent
  • Algorithm-grounded pricing replaced manual guesswork
  • Commercialization: System now sold as Quotely, a standalone SaaS product to other operators
Key Insight

AI systems succeed when they reduce operational ambiguity, not when they maximize sophistication. Focused on structured information capture and reliable handoffs over open-ended interaction.

Scenario-First Energy Modeling: Domain Core + Multi-Engine Adapters

Project Epsilon • Energy systems · Infrastructure • Originated from MBA deployability models
Expand full case study
Challenge

Clean-energy planning slows down when sizing, stress-testing, and finance run in disconnected tools: assumptions drift, comparisons break, and nobody can defend a single narrative from deck to data room. During my MBA I built spreadsheet models to reason about deployability; they surfaced the questions but stayed siloed.

Engineering risk: Naively porting workbook logic to a UI yields surface parity without semantic correctness (hidden unit mismatches, unsafe handoffs between sizing and stress, economics that ignore feeder reality).

Solution

Project Epsilon connects those worlds with one version-controlled Scenario that feeds every engine—no copy-paste drift between models.

  • Six-step guided workflow: define scenario → size portfolio → stress-test → evaluate finance → validate network → generate report
  • Three production adapters (today): DC-PPA · ISO-RTO · Microgrid, plus network validation (feeder / N-1 / AC power flow) as the physical feasibility layer
  • Explicit handoffs: typed CapacitySeed from sizing into stress and finance
  • Assurance layer: structured audit findings, 100-point confidence scoring with severity-based deductions, SHA-256 scenario-hash staleness, prerequisite gating so stress cannot run on silent stale inputs
  • BOM bridge: sizing flows emit starter equipment catalog rows (procurement-grade narrative from modeled MW/MWh)
  • Reporting: compare runs, field-by-field assumption diffs, Markdown/HTML export

Positioning: Planning and diligence for data centers, microgrids, and RTO-style stress—not a real-time EMS.

Technical Stack

Python: custom PyPSA linear programs for 24/7 CFE and microgrid sizing; ISO-RTO Monte Carlo stress testing; agent-based VPP dispatch; AC power flow / distribution validation. Typed Scenario contract, EngineRegistry, ScenarioStore, normalized finance assumptions (e.g. WACC, battery efficiency) for cross-engine comparability.

Delivery: Public app hosted on Cloudflare Workers (epsilon.xblavania.workers.dev).

My Role

Product builder and engineer: learned distributed-energy dynamics deeply enough to shape module boundaries (what must be true for sizing, stress, finance, and network to tell one coherent story). Implemented custom PyPSA LPs, ABM-style VPP dispatch, Monte Carlo stress paths, and end-to-end architecture that bridges physical network validation with financial planning. Owned scenario contract, adapter patterns, and assurance story—not just a spreadsheet port.

Results
Live platform: epsilon.xblavania.workers.dev
  • Single scenario state across engines; field-by-field diffs and run comparison
  • Structured audit + confidence and stale-run detection tied to scenario hashes
  • At a glance: 6 workflow steps · 3 production adapters · 100-point confidence model · 11 starter BOM SKUs

Outputs are for analysis and professional judgment only—not legal, regulatory, or FERC filing advice.

Key Insight

Each workflow step surfaces a risk class the others cannot see. Siloed software breaks narrative coherence between pitch physics and finance; deployability is a coupling problem. The product’s job is to make those couplings explicit, comparable, and reviewable—not to maximize chart polish.

85% False Alert Reduction: Student Safety Detection System

Netsweeper (EdTech) • 2022 to 2023 • 35-school district deployment
Expand full case study
Challenge

Student safety monitoring product generated excessive false alerts, creating alert fatigue and reducing operational trust. Staff stopped relying on the system.

Solution

Reframed the problem from "detection accuracy" to "operational trust":

  • Prioritized signal quality over feature expansion
  • Collaborated with detection model team on tuning
  • Redesigned workflow to handle edge cases better
  • Made operational trust a core product metric
My Role

Product Manager. Drove problem definition, stakeholder alignment, and cross-functional prioritization around detection quality.

Results
False alerts reduced by 85%
  • Improved operational usability for school staff
  • Increased trust in system outputs
  • Reduced alert fatigue
Key Insight

Detection systems fail when users stop trusting the signal. Accuracy metrics alone are insufficient. Operational reliability matters more than theoretical performance.

My Approach

  • Build for operational simplicity first, add features only when they solve real problems
  • Ship AI that reduces friction in existing workflows, not AI for novelty's sake
  • Fix decision-making upstream before building dashboards downstream
  • Prioritize signal quality over data volume
  • Design products that reduce cognitive load, not increase it
